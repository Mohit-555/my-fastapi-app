# Frontend to Backend API Mapping Guide

This guide maps each page of the RDPMS Frontend interface (corresponding to the screenshots in `frontend_png/`) to its respective FastAPI backend HTTP and WebSocket API endpoints.

---

## 1. Main Dashboard View
**File**: `screencapture-3-6-93-103-3001-dashboard-2026-07-09-11_30_12.png`

The central overview displaying active alert tallies, active assets count, station statuses, and geographical division filtering.

### Mapped Backend APIs:
* **`GET /dashboard/summary`** (Defined in `app/routers/dashboard.py`)
  * **Description**: Returns general metrics, including overall station health breakdown, counts of active critical vs. warning alerts, and zone-level summaries.
  * **Response Schema**:
    ```json
    {
      "total_stations": 12,
      "healthy_stations": 10,
      "degraded_stations": 2,
      "active_alerts": { "critical": 3, "warning": 7 }
    }
    ```
* **`GET /dashboard/asset-status`** (Defined in `app/routers/dashboard.py`)
  * **Description**: Delivers real-time status counts categorized by asset class (Point Machine, Track Circuit, Signal, IPS).
* **`GET /zones/list`**, **`GET /divisions/list`**, **`GET /stations/list`**
  * **Description**: Drives the hierarchical location selector dropdown filters in the top navbar.

## 2. Live Alerts Feed
**File**: `screencapture-3-6-93-103-3001-alerts-live-2026-07-09-11_31_24.png`

Real-time notification list monitoring active failures and predictive diagnostic warnings.

### Mapped Backend APIs:
* **`WS /ws/alerts/{station_code}`** (Defined in `app/routers/websocket.py`)
  * **Description**: Permanent WebSocket channel pushing active alert payloads directly to the feed the instant they are generated by the Rule Engine.
* **`GET /alerts/active`** (Defined in `app/routers/alerts.py`)
  * **Description**: HTTP fallback/initial fetch endpoint to load the currently unresolved alert records on page refresh.
  * **Query Params**: `station_id` (optional), `asset_type_hex` (optional).
* **`PUT /alerts/{alert_id}/acknowledge`** (Defined in `app/routers/alerts.py`)
  * **Description**: Triggered when a console operator clicks the "Acknowledge" button.
* **`PUT /alerts/{alert_id}/clear`** (Defined in `app/routers/alerts.py`)
  * **Description**: Invoked when an operator logs comments and clears an alert manually.

## 3. Alerts Analytics & Summary
**File**: `alert_summery.png`

Analytical screen showcasing charts of alert frequency, category analysis, and distribution by equipment type.

### Mapped Backend APIs:
* **`GET /statistics/alerts/summary`** (Defined in `app/routers/statistics.py`)
  * **Description**: Returns alert statistics grouped by date, category (Failure, Predictive), and asset type for graph rendering.
  * **Query Params**: `start_date`, `end_date`, `station_id`.
* **`GET /alerts/list`** (Defined in `app/routers/alerts.py`)
  * **Description**: Fetches list of alerts based on analytical filters.

---

## 4. Alert History log
**File**: `alert_history.png`

Searchable log history containing all resolved, closed, and archived system alert records.

### Mapped Backend APIs:
* **`GET /alerts/history`** (Defined in `app/routers/alerts.py`)
  * **Description**: Returns a paginated list of historically cleared alerts.
  * **Query Params**: `page`, `limit`, `station_id`, `start_date`, `end_date`, `search`.
* **`GET /alerts/{alert_id}/details`** (Defined in `app/routers/alerts.py`)
  * **Description**: Opens a sidebar details pane showing diagnostic comments, actions taken, and sensor logs leading up to the trigger.

---

## 5. Live Telemetry Monitor
**File**: `telemetry_live.png`

Dynamic dashboard detailing real-time sensor parameters (voltages, current waveforms) updating live.

### Mapped Backend APIs:
* **`WS /ws/telemetry/{station_code}`** (Defined in `app/routers/websocket.py`)
  * **Description**: Streams incoming gateway packet readings.
* **`GET /realtime/current-state/{station_id}`** (Defined in `app/routers/realtime.py`)
  * **Description**: Populates the UI widgets with the last-cached values immediately upon loading the page before WebSocket stream initiates.

---

## 6. Historical Telemetry Charts
**File**: `telemetry_history.png`

Time-series graphs visualizing sensor trends against configured warning and critical limits.

### Mapped Backend APIs:
* **`GET /telemetry/history`** (Defined in `app/routers/telemetry.py`)
  * **Description**: Queries historical database values to render telemetry trend graphs.
  * **Query Params**: `para_id`, `start_time`, `end_time`.
* **`GET /assets/thresholds/resolve/{asset_type_hex}/{parameter_type_hex}`** (Defined in `app/routers/assets.py`)
  * **Description**: Fetches standard normal/warning/critical levels to draw boundary lines on the charts.

---

## 7. Asset Details & Inventory Profile
**File**: `asset_detail.png`

A profile page for an individual hardware asset (e.g., a specific Point Machine at a station).

### Mapped Backend APIs:
* **`GET /assets/{asset_id}`** (Defined in `app/routers/assets.py`)
  * **Description**: Retrieves asset profile metadata (brand, vendor name, installation date, serial code).
* **`GET /assets/{asset_id}/parameters`** (Defined in `app/routers/assets.py`)
  * **Description**: Returns all parameter sensors mapped to this specific asset, showing current values and locations (`prloc`).

### Mapped Backend APIs:

## 9. Equipment Room (Relay Room) Environment
**Files**: `equipment_roomlive.png` and `equipment_room_history.png`

Real-time and historical views for ambient conditions (temperature, humidity, fire sensors) in the station relay room.

### Mapped Backend APIs:
* **`GET /equipment-room/live`** (Defined in `app/routers/equipment_room.py`)
  * **Description**: Returns live environmental statuses for a station's relay room.
* **`GET /equipment-room/history`** (Defined in `app/routers/equipment_room.py`)
  * **Description**: Retrieves historical logs of temperature and humidity readings.

## 10. System Health & Gateway Monitoring
**Files**: `health_live.png` and `health_summary.png`

Checks status of hardware gateways (online/offline), signal strengths, and network health.

### Mapped Backend APIs:
* **`WS /ws/health/{station_code}`** (Defined in `app/routers/websocket.py`)
  * **Description**: Pushes real-time hardware status changes (gateway online/offline, fault logs).
* **`GET /gateway/list`** (Defined in `app/routers/gateway.py`)
  * **Description**: Lists registered gateway hardware details (IMEI, associated station).

---

## 11. Maintenance Mode Manager
**File**: `maintainance.png`

Controls active maintenance windows, letting field technicians mute alert alarms.

### Mapped Backend APIs:
* **`GET /maintenance/active`** (Defined in `app/routers/maintenance.py`)
  * **Description**: Lists assets currently set in maintenance mode.
* **`POST /maintenance/start`** (Defined in `app/routers/maintenance.py`)
  * **Description**: Starts a maintenance window.
  * **Payload**: `{ "asset_id": 4, "duration_minutes": 60 }`
* **`POST /maintenance/stop`** (Defined in `app/routers/maintenance.py`)
  * **Description**: Ends a maintenance window early.

---

## 12. Backend Performance Dashboard
**File**: `performance.png`

Displays API latency, system memory usage, database connection status, and event queue backlogs.

### Mapped Backend APIs:
* **`GET /metrics`** (Defined in `app/main.py`)
  * **Description**: Exposes Prometheus-format system metrics.
* **`GET /api/monitoring/health`** (Defined in `app/routers/monitoring.py`)
  * **Description**: Queries status of connected components (Database, Redis, WebSocket manager, Scheduler, Alert Processor).
